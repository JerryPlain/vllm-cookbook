# 仓库导读（从“能跑”到“可复用”）

本文解释这个 `vllm-cookbook` 的目标、结构、建议使用顺序，以及每个文件承担的职责。

## 1. 这个仓库解决什么问题

你未来再用 vLLM 时，通常会遇到三类问题：

1. 环境与参数不一致，导致“上次能跑这次不能跑”。
2. 模型并行/显存/上下文参数组合不稳定，排障成本高。
3. LoRA 适配流程容易散落在聊天记录里，无法复用。

这个仓库的设计是：
- 用 `examples/` 保证每个知识点都有可运行脚本。
- 用 `src/vllm_cookbook/` 固化可复用的 fail-fast 工具。
- 用 `docs/zh/` 沉淀原理与排障方法，减少“只记结论不懂原因”。

## 2. 目录结构与职责

- `examples/`
  - 面向“直接运行”的入口脚本。
  - 每个脚本只演示一个核心能力，便于定位问题。
- `scripts/`
  - 提供统一环境变量和运行方式，降低命令错误。
- `src/vllm_cookbook/`
  - 放通用函数，比如 TP 合法性检查。
  - 原则：能沉淀就不要重复写在脚本里。
- `docs/`
  - 英文简版安装/坑点速查。
- `docs/zh/`
  - 中文深度解释：原理、参数、LoRA、实战建议。
- `tests/`
  - 轻量测试，防止基础工具函数被改坏。

## 3. 建议阅读与使用顺序

1. 先跑通 `scripts/run_minimal.sh`。
2. 看 `examples/04_tp_check_heads.py` 理解 TP 合法性。
3. 再看 `examples/03_lora_request.py` 学会 LoRA 运行时注入。
4. 遇到性能/显存问题时查 `docs/zh/03_参数与调优清单.md`。

## 4. 这个仓库的“方法论”

- 先做可复现最小闭环，再做复杂优化。
- 先做 fail-fast 检查，再启动重任务。
- 文档必须对应可运行脚本，避免“只讲概念”。

## 5. 你后续可以扩展什么

- 增加多模型 benchmark 脚本（吞吐、延迟、显存）。
- 增加 OpenAI-compatible server 调用示例。
- 增加多 LoRA 热切换示例与并发场景验证。
