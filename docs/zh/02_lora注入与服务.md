# LoRA 注入与服务化实践

本文解释这个仓库里的 LoRA 使用方式，以及它和“合并权重”路线的差异。

## 1. LoRA 的基本思想

LoRA 通过低秩增量矩阵对基座模型做参数高效适配。推理时你有两种主流路线：

1. 合并路线：把 LoRA 合入基座权重，得到一个新模型目录。
2. 注入路线：保留基座模型，运行时按请求附加 LoRA。

本仓库示例采用注入路线（`LoRARequest`），更适合 cookbook 和多适配器服务。

## 2. `LoRARequest` 在脚本里的工作流

`examples/03_lora_request.py` 的流程：

1. 读取 `MODEL`、`TP`、`LORA_PATH`。
2. 先执行 `assert_tp_valid()` 做并行合法性检查。
3. 初始化 `LLM(..., enable_lora=True)`。
4. 如果 `LORA_PATH` 非空，构造 `LoRARequest` 并传给 `generate()`。
5. 如果 `LORA_PATH` 为空，则退化为基座模型推理。

这种写法有两个优点：
- 一个脚本同时覆盖“有 LoRA / 无 LoRA”两种路径。
- 配置错误会尽早暴露。

## 3. LoRA 注入常见问题

- 路径问题：`LORA_PATH` 指向错误目录或权限不足。
- 兼容问题：LoRA 与基座模型结构不匹配。
- 并行问题：TP 设置不合法或显存不足。

排查顺序建议：
1. 不带 LoRA 先跑通基座模型。
2. 带 LoRA 且 TP=1 验证最小闭环。
3. 再逐步提升 TP 与并发。

## 4. 何时选择“合并权重”

你可以在以下场景考虑合并路线：
- 只服务单一 LoRA，且长期固定。
- 希望部署链路更简单。

但如果你要频繁切换适配器，运行时注入通常更灵活。
